Introduction

Event Sourcing - quite young and modern approach for building data storage and access architecture. One of the first description of this approach is made by Martin Fowler in his article, a well known author of the books and articles related to software development. You can also find various web-resources related to Event Sourcing, in the Internet. However, as Event Sourcing and CQRS are closly related to Heeus Project, we (Heeus authors) decided to publish this review article describing the Event Sourcing architecture briefly.

What is Event Sourcing

Up to the present times the classical CRUD was the dominating approach for building data storage and access architecture. CRUD is the abbreviation for Create-Read-Update-Delete, as the names of basic data access operations. This architecture assumes that the database predominantly stores the final states of the objects. To say simple, once the object is being modified by series of update procedures, the database only stores its state which corresponds to the final update.

In contrast to CRUD, Event Souring declares the opposite aproach: every action which creates or modifies any object, consistently stored in database as the “Events”. This produces so called “Event Log”. The only way to see the state of any object for certain date and time, just as the state of the compelte system, is calculation, which literally means walking through the Event Log and applying Events, one by one. This way of operating with the data is much more complicated in development than classical CRUD, however it finally provides number of undeniable advantages.

What is CQRS

CQRS - is the abbreviation to Command Query Responsibility Segregation, which literally means the responsibility segregation to commands and queries. This development approach applied to Event Sourcing means that read operations do not operate directly to the Event Log, but addressed to the special storage, which is build on the base of processing and calculation over the Event Log, and which is specially optimized for read operations. This improves system performance, scalability and security.

Why to use Event Sourcing and CQRS

Imagine that we have the platform, which uses Event Sourcing + CQRS architecture, and allows deploying and running user microservices. What advantages this architecture provides for running those microservices and user apps?

  • The complete system is much more predictable and scalable.
  • The data storage is based on the Event Log, which is superb for any solutions requiring an Audit Log: online stores, IoT (internet of things), Hotel/restaurant/cafe, warehouse, issue and project management, etc.
  • The system is more fault tolerant.
  • CQRS-based system has potentially higher performance, even on the relatively poor hardware.
  • User apps and microservices, operating with the data based on CQRS, much more easy to test and debug.
